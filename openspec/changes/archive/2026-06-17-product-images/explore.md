# Exploration: product-images — Multi-Image Carousel per Product

> SDD EXPLORE phase. Investigation only — no code, no proposal.
> Artifact store: hybrid (engram `sdd/product-images/explore` #380 + this file).
> Date: 2026-06-17.

## Intent

Each product accepts **N images** (uploaded by an admin), stored by the **backend on local filesystem** (academic scope — ephemeral-in-container caveat accepted). The frontend shows them in a **carousel** like real e-commerce. The DB stores only **references (paths)**, never binaries.

Agreed framing: the catalog **list** shows ONE **cover** per product; the **detail** shows the full **carousel** (N). This drives both the data model and list performance.

## Current State

**Backend** (Spring Boot 3.3.5, Java 17, PostgreSQL 16):
- `Product` entity has a single `String imageUrl` (nullable, `VARCHAR(500)` in V1). No image relation exists.
- `ProductRequest`/`ProductResponse` carry `imageUrl` as a plain string; `ProductMapper.toResponse()` maps it directly.
- `AdminProductController` (`/api/admin/products`) accepts only `@RequestBody` JSON. No multipart anywhere.
- `ProductController` (`/api/products`) returns `ProductResponse` for both list and detail.
- `ProductServiceImpl.search()` maps `Page<Product>` via `productMapper::toResponse` (single query, category JOIN). `getById()` same mapper.
- Migrations V1–V4 applied. Next = **V5**.
- `SecurityConfig`: `/api/admin/**` requires ROLE_ADMIN; `GET /api/products/**` is `permitAll`. No static resource handler. `CorsConfig` covers `/**`.

**Frontend** (Angular 18, standalone, Jest 29):
- `ProductResponse.imageUrl: string | null` with `PLACEHOLDER_IMAGE` fallback.
- `product-card`: uses `product.imageUrl ?? PLACEHOLDER_IMAGE` — no change needed for list.
- `product-detail`: single `<img>`, no carousel — must be upgraded.
- `product.service`: no upload capability.

## Approaches & Recommendations

**1. Serving strategy** — Streaming `@RestController` (`/api/uploads/images/{filename}`) **[recommended]** over Spring static resource handler. Fits the layered architecture, no extra MVC config, fully testable with `@WebMvcTest`, explicit path-traversal guard.

**2. Existing `products.image_url`** — Keep as **denormalized cover** **[recommended]**. List query stays single-table; card unchanged; idiomatic with the project's existing denormalization (cached `products.stock` + kardex, price snapshot). Synced atomically on cover change/delete inside `@Transactional`. (Alternatives: drop + JOIN, or drop + separate cover query — both worse for list performance/compat.)

**3. DTO strategy** — Same `ProductResponse` record with `images` field, `@JsonInclude(NON_NULL)` so it's null in list, populated in `getById()` **[recommended]** over a separate `ProductDetailResponse`. Simpler mapper evolution; null-in-list acceptable at academic scope.

**4. Upload granularity** — **Single upload** (`POST /api/admin/products/{id}/images`) **[recommended]** over batch. Simpler validation/error reporting, simpler `MockMultipartFile` tests, per-file frontend progress.

## Data Model (V5)

```sql
CREATE TABLE product_image (
    id            BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    product_id    BIGINT       NOT NULL REFERENCES products (id),
    path          VARCHAR(500) NOT NULL,
    display_order SMALLINT     NOT NULL DEFAULT 0,
    is_cover      BOOLEAN      NOT NULL DEFAULT FALSE,
    created_at    TIMESTAMPTZ  NOT NULL DEFAULT now()
);
CREATE INDEX idx_product_image_product ON product_image (product_id);
CREATE UNIQUE INDEX idx_product_image_cover ON product_image (product_id) WHERE is_cover = TRUE;
```

`products.image_url` remains. On first image upload → set as cover → update `products.image_url` to the serving URL, atomically within `@Transactional`.

## Upload API

```
POST   /api/admin/products/{id}/images               Upload one image (multipart/form-data)
DELETE /api/admin/products/{id}/images/{imgId}        Delete image
PATCH  /api/admin/products/{id}/images/reorder        Reorder [{id, displayOrder}]
PATCH  /api/admin/products/{id}/images/{imgId}/cover  Promote to cover
GET    /api/uploads/images/{filename}                 Serve binary (public, permitAll)
```

Validation: content-type allowlist (jpeg/png/webp), max 5 MB, max 10 images per product, UUID filename on disk.

## Risks

1. Path traversal on `ImageServeController` — sanitize filename, check `path.startsWith(uploadDir)`.
2. Ephemeral uploads dir in containers — `@TempDir` in integration tests; document caveat.
3. `products.image_url` sync atomicity on cover change/delete (inside `@Transactional`).
4. `product.getImages()` lazy collection — access within open `@Transactional(readOnly=true)` in `getById()`.
5. `@WebMvcTest` multipart setup is non-trivial — `MockMultipartFile` pattern must be documented in tasks.
