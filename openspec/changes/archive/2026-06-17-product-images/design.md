# Design: product-images — Multi-Image Carousel per Product

> SDD DESIGN phase. The technical HOW: components, contracts, algorithms, testing strategy.
> Reads PROPOSAL (`sdd/product-images/proposal` #382) + EXPLORE (`sdd/product-images/explore` #380).
> Artifact store: hybrid (engram `sdd/product-images/design` + this file).
> Date: 2026-06-17. All architectural decisions are LOCKED in the proposal — this designs the implementation.

## 1. Architecture at a glance

Layered, interface-first, matching the existing project: `controller → service (interface + impl) → repository → model`. Controllers never touch repositories; entities never leave the service. Two new vertical slices plus surgical edits to the product read path.

```
            ┌──────────────────────────── WEB ────────────────────────────┐
            │                                                              │
ADMIN  ───▶ AdminProductImageController  (multipart POST/DELETE/PATCH, ROLE_ADMIN)
PUBLIC ───▶ ImageServeController          (GET binary, permitAll)
PUBLIC ───▶ ProductController.getById     (existing — now carries `images`)
            │                                                              │
            ▼                              ▼                               ▼
        ProductImageService          StorageService                  ProductService
        (interface + impl)           (interface + impl)              (existing, getById edit)
            │           │                  │                              │
            ▼           ▼                  ▼                              ▼
   ProductImageRepository  ProductRepository   LocalStorageServiceImpl   ProductMapper
            │                                  (filesystem I/O)          (toResponseWithImages)
            ▼
   product_image table (V5)  ◀── FK ──▶  products.image_url (denormalized cover, synced in @Transactional)
```

Key seam for Strict TDD: **`StorageService` isolates ALL filesystem I/O**. `ProductImageServiceImpl` depends on the interface, so its unit test mocks storage and asserts orchestration + cover algorithm without ever touching the disk. The real `LocalStorageServiceImpl` is covered only by the integration test (real `@TempDir`).

## 2. Data layer

### 2.1 Migration `V5__add_product_image.sql`

Matches V1 conventions exactly: `BIGINT GENERATED BY DEFAULT AS IDENTITY`, `TIMESTAMPTZ DEFAULT now()`, explicit `REFERENCES`, named indexes.

```sql
-- V5: imágenes por producto (galería + carrusel). Binarios en filesystem; aquí solo refs.

CREATE TABLE product_image (
    id            BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    product_id    BIGINT       NOT NULL REFERENCES products (id),
    path          VARCHAR(500) NOT NULL,            -- filename UUID en disco (host-agnostic)
    display_order SMALLINT     NOT NULL DEFAULT 0,
    is_cover      BOOLEAN      NOT NULL DEFAULT FALSE,
    created_at    TIMESTAMPTZ  NOT NULL DEFAULT now()
);

CREATE INDEX idx_product_image_product ON product_image (product_id);

-- A lo sumo UNA portada por producto. Índice único parcial = la garantía vive en la DB,
-- la sincronización de products.image_url vive en la app (@Transactional).
CREATE UNIQUE INDEX idx_product_image_cover ON product_image (product_id) WHERE is_cover = TRUE;
```

NO max-images CHECK constraint — row-count limits are not portable as DB constraints and the 400 mapping belongs in the service (locked decision Q5). `products.image_url` stays untouched by the migration (kept as denormalized cover, synced by the service).

### 2.2 Entity `model/ProductImage.java`

Same Lombok style as `Product` (`@Getter @Setter @NoArgsConstructor`, field access). `product` is `@ManyToOne(LAZY)`.

```java
@Entity
@Table(name = "product_image")
@Getter @Setter @NoArgsConstructor
public class ProductImage {
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @ManyToOne(fetch = FetchType.LAZY, optional = false)
    @JoinColumn(name = "product_id")
    private Product product;

    @Column(nullable = false, length = 500)
    private String path;            // UUID filename on disk

    @Column(name = "display_order", nullable = false)
    private short displayOrder;

    @Column(name = "is_cover", nullable = false)
    private boolean cover;

    @Column(name = "created_at", insertable = false, updatable = false)
    private OffsetDateTime createdAt;   // DB-defaulted
}
```

`createdAt` is `insertable=false, updatable=false` so Hibernate reads the DB `now()` default and `ddl-auto: validate` still passes (TIMESTAMPTZ → `OffsetDateTime`).

### 2.3 `Product` modification — LAZY one-to-many

```java
@OneToMany(mappedBy = "product", fetch = FetchType.LAZY)
@OrderBy("displayOrder ASC, id ASC")
private List<ProductImage> images = new ArrayList<>();
```

- LAZY is mandatory: `search()` (list) must NOT load images. With `open-in-view: false` (confirmed in `application.yml`), the collection only materializes inside an open transaction — exactly why `getById()` must stay `@Transactional(readOnly = true)` (see §5).
- `@OrderBy` gives deterministic carousel order at the JPA level (cover is also `displayOrder=0` by convention, so it lands first).
- NO cascade: image rows are managed explicitly by the service. Product soft-delete keeps image rows + files (locked Q3 — orphans accepted/documented).

### 2.4 Repository `repository/ProductImageRepository.java`

```java
public interface ProductImageRepository extends JpaRepository<ProductImage, Long> {
    List<ProductImage> findByProductIdOrderByDisplayOrderAscIdAsc(Long productId);
    long countByProductId(Long productId);
    Optional<ProductImage> findByProductIdAndCover(Long productId, boolean cover);
    Optional<ProductImage> findFirstByProductIdAndIdNotOrderByDisplayOrderAscIdAsc(Long productId, Long excludedId);
}
```

- `countByProductId` → enforces MAX_IMAGES (10) in the service.
- `findByProductIdAndCover(id, true)` → current cover (used on set-cover demotion).
- `findFirstBy...IdNot...OrderBy...` → the **cover-promotion candidate** when the current cover is deleted (lowest `displayOrder`, tie-break by `id`).

## 3. Storage abstraction

### 3.1 Interface `service/StorageService.java`

```java
public interface StorageService {
    /** Persists the binary, returns the generated filename (UUID + ext). Never the full path. */
    String store(MultipartFile file);
    /** Loads the binary as a Spring Resource for streaming. Throws if absent. */
    Resource load(String filename);
    /** Physically removes the binary. No-op if already gone. */
    void delete(String filename);
}
```

The contract returns/accepts **filenames only** (never absolute paths) — the path-traversal guard lives entirely inside the impl, so callers cannot bypass it.

### 3.2 Impl `service/impl/LocalStorageServiceImpl.java`

```java
@Service
public class LocalStorageServiceImpl implements StorageService {

    private final Path uploadDir;

    public LocalStorageServiceImpl(@Value("${app.uploads.dir}") String dir) {
        this.uploadDir = Paths.get(dir).toAbsolutePath().normalize();
    }

    @PostConstruct
    void init() throws IOException { Files.createDirectories(uploadDir); }

    @Override
    public String store(MultipartFile file) {
        String ext = extensionFor(file.getContentType());          // jpg|png|webp from MIME, NOT client name
        String filename = UUID.randomUUID() + "." + ext;
        Path target = resolveSafe(filename);
        try (InputStream in = file.getInputStream()) {
            Files.copy(in, target, StandardCopyOption.REPLACE_EXISTING);
        } catch (IOException e) { throw new StorageException("No se pudo guardar la imagen", e); }
        return filename;
    }

    @Override
    public Resource load(String filename) {
        Path file = resolveSafe(filename);                          // guard runs here too
        Resource res = new FileSystemResource(file);
        if (!res.exists() || !res.isReadable())
            throw new ResourceNotFoundException("Imagen no encontrada: " + filename);
        return res;
    }

    @Override
    public void delete(String filename) {
        try { Files.deleteIfExists(resolveSafe(filename)); }
        catch (IOException e) { throw new StorageException("No se pudo borrar la imagen", e); }
    }

    /** PATH-TRAVERSAL GUARD — the single CRITICAL security control of this change. */
    private Path resolveSafe(String filename) {
        Path resolved = uploadDir.resolve(filename).normalize();
        if (!resolved.startsWith(uploadDir)) {                      // canonicalize + containment assertion
            throw new IllegalArgumentException("Ruta de archivo inválida: " + filename);
        }
        return resolved;
    }
}
```

Guard rationale: `uploadDir` is normalized+absolute once in the constructor. Every `resolveSafe` normalizes the candidate and asserts `startsWith(uploadDir)`. `../../etc/passwd` resolves outside `uploadDir` → `IllegalArgumentException` → mapped to **400** by the existing `GlobalExceptionHandler.handleIllegalArgument` (already present, line 87). No new handler needed for traversal.

Extension is derived from the validated MIME type (`image/jpeg→jpg`, `image/png→png`, `image/webp→webp`), never from the client-supplied filename — closes a second injection vector.

`StorageException` is a new `RuntimeException` (I/O failure) → maps to 500 (falls through to Spring default; not a client error). Place in `exception/StorageException.java`.

## 4. Service layer

### 4.1 Constants & validation (in `ProductImageServiceImpl`)

```java
static final int  MAX_IMAGES_PER_PRODUCT = 10;
static final long MAX_FILE_SIZE_BYTES    = 5L * 1024 * 1024;           // 5 MB
static final Set<String> ALLOWED_TYPES   = Set.of("image/jpeg", "image/png", "image/webp");
```

Validation order in `upload` (fail fast, cheapest first):
1. content-type ∈ ALLOWED_TYPES → else `IllegalArgumentException` (400).
2. `file.getSize() <= MAX_FILE_SIZE_BYTES` → else `IllegalArgumentException` (400). (Spring's multipart `max-file-size` is the outer guard at 413; this app-level check is the inner net so the service stays self-contained and unit-testable.)
3. product exists → else `ResourceNotFoundException` (404).
4. `countByProductId < MAX` → else `IllegalArgumentException` (400) "máximo 10 imágenes".

Only after all checks pass do we call `storage.store(...)` — never write a file we'd have to roll back.

### 4.2 Interface `service/ProductImageService.java`

```java
public interface ProductImageService {
    ProductImageResponse upload(Long productId, MultipartFile file);
    void delete(Long productId, Long imageId);
    List<ProductImageResponse> reorder(Long productId, List<ImageOrderRequest> order);
    ProductImageResponse setCover(Long productId, Long imageId);
}
```

All four methods are `@Transactional` on the impl (NOT readOnly — they mutate). Every cover-touching method does its DB mutations + `products.image_url` sync inside the SAME transaction, so the partial-unique cover index never sees two covers and `image_url` never diverges.

### 4.3 Cover-promotion algorithm (the heart of the change)

The invariant: **a product with ≥1 image has exactly one cover; `products.image_url` always reflects that cover's serving URL (or null when no images).**

`upload(productId, file)`:
```
validate (§4.1)
filename = storage.store(file)
img = new ProductImage(product, filename, displayOrder = countBefore, cover = (countBefore == 0))
save(img)
if img.cover:                       # first image of the product
    product.imageUrl = serveUrl(filename)   # sync denormalized cover
return map(img)
```

`delete(productId, imageId)`:
```
img = findOrThrow(imageId) within product            # 404 if absent / wrong product
wasCover = img.cover
storage.delete(img.path)                              # physical file removed (locked Q3: only on explicit delete)
repository.delete(img)
if wasCover:
    next = repo.findFirstByProductIdAndIdNotOrderByDisplayOrderAscIdAsc(productId, imageId)
    if next present:
        next.cover = true
        product.imageUrl = serveUrl(next.path)        # PROMOTE lowest-order survivor
    else:
        product.imageUrl = null                       # last image deleted → null (frontend → PLACEHOLDER)
# deleting a NON-cover image leaves cover + image_url untouched
```

`setCover(productId, imageId)`:
```
target = findOrThrow(imageId) within product
current = repo.findByProductIdAndCover(productId, true)
if current present and current.id != target.id:
    current.cover = false                             # demote FIRST (avoid transient 2-cover state)
target.cover = true
product.imageUrl = serveUrl(target.path)
```
Demote-before-promote ordering matters: the partial-unique index forbids two `is_cover=TRUE` rows. Within one transaction Hibernate flushes in order; to be safe the impl sets `current.cover=false`, flushes (or relies on same-tx ordering), then `target.cover=true`. If a flush-order race is observed in integration, add an explicit `repository.flush()` between demote and promote — documented as the fallback.

`reorder(productId, order)`:
```
images = repo.findByProductIdOrderBy...(productId)
index by id
for each {id, displayOrder} in request:
    require id belongs to product                     # else 400
    image.displayOrder = displayOrder
# reorder does NOT change which image is cover, NOR image_url
return images sorted by new displayOrder
```
Reorder is intentionally cover-neutral: changing display order never re-elects the cover. Cover is moved only by `setCover` / `delete`. (If product owner wants the new first image as cover, they call setCover explicitly — keeps the two concerns orthogonal.)

`serveUrl(filename)` = `baseUrl + "/api/uploads/images/" + filename`, where `baseUrl` is `@Value("${app.uploads.base-url:http://localhost:8080}")`. This is the SINGLE place the URL is composed in the write path; the mapper composes it in the read path (§6) — both share the same `base-url` property.

## 5. Web layer

### 5.1 `controller/AdminProductImageController.java` (ROLE_ADMIN via `/api/admin/**`)

```java
@RestController
@RequestMapping("/api/admin/products/{productId}/images")
public class AdminProductImageController {
    private final ProductImageService service;   // constructor injection

    @PostMapping(consumes = MediaType.MULTIPART_FORM_DATA_VALUE)
    @ResponseStatus(HttpStatus.CREATED)
    public ProductImageResponse upload(@PathVariable Long productId,
                                       @RequestParam("file") MultipartFile file) {
        return service.upload(productId, file);
    }

    @DeleteMapping("/{imageId}")
    @ResponseStatus(HttpStatus.NO_CONTENT)
    public void delete(@PathVariable Long productId, @PathVariable Long imageId) {
        service.delete(productId, imageId);
    }

    @PatchMapping("/reorder")
    public List<ProductImageResponse> reorder(@PathVariable Long productId,
                                              @Valid @RequestBody List<ImageOrderRequest> order) {
        return service.reorder(productId, order);
    }

    @PatchMapping("/{imageId}/cover")
    public ProductImageResponse setCover(@PathVariable Long productId, @PathVariable Long imageId) {
        return service.setCover(productId, imageId);
    }
}
```

`ImageOrderRequest` = new record `dto/request/ImageOrderRequest.java`: `(@NotNull Long id, @NotNull Short displayOrder)`.

Authorization is INHERITED — the path is under `/api/admin/**`, already `hasRole("ADMIN")` in `SecurityConfig` line 59. No new security rule for admin endpoints.

### 5.2 `controller/ImageServeController.java` (public, permitAll)

```java
@RestController
@RequestMapping("/api/uploads/images")
public class ImageServeController {
    private final StorageService storage;

    @GetMapping("/{filename}")
    public ResponseEntity<Resource> serve(@PathVariable String filename) {
        Resource res = storage.load(filename);                 // guard inside load() → IllegalArgument → 400
        return ResponseEntity.ok()
                .contentType(MediaTypeFactory.getMediaType(res).orElse(MediaType.APPLICATION_OCTET_STREAM))
                .cacheControl(CacheControl.maxAge(7, TimeUnit.DAYS).cachePublic())
                .body(res);
    }
}
```

Streaming `ResponseEntity<Resource>` (Spring streams `FileSystemResource` without loading into heap). Content-Type inferred from filename extension via `MediaTypeFactory`. 7-day public cache (images are immutable — UUID filenames never change). Missing file → `ResourceNotFoundException` (from `load`) → 404. Traversal filename → `IllegalArgumentException` → 400.

### 6.2 `dto/response/ProductResponse.java` modification

Add ONE field at the end, with FIELD-level `@JsonInclude(NON_NULL)` (locked Q4 — only `images` is conditionally present):


### 6.3 `mapper/ProductMapper.java` modification — TWO mapping paths

```java
@Component
public class ProductMapper {
    private final String baseUrl;
    public ProductMapper(@Value("${app.uploads.base-url:http://localhost:8080}") String baseUrl) {
        this.baseUrl = baseUrl;
    }

    /** LEAN — list/create/update. images = null → omitted. NO lazy access. */
    public ProductResponse toResponse(Product p) {
        return new ProductResponse(p.getId(), p.getSku(), p.getName(), p.getDescription(),
                p.getPrice(), p.getStock(), p.getImageUrl(), p.isActive(),
                p.getCategory().getId(), p.getCategory().getName(), null);
    }

    /** RICH — detail only. Reads p.getImages() → REQUIRES open transaction. */
    public ProductResponse toResponseWithImages(Product p) {
        List<ProductImageResponse> imgs = p.getImages().stream()
                .map(i -> new ProductImageResponse(i.getId(),
                        baseUrl + "/api/uploads/images/" + i.getPath(),
                        i.getDisplayOrder(), i.isCover()))
                .toList();
        return new ProductResponse(p.getId(), p.getSku(), p.getName(), p.getDescription(),
                p.getPrice(), p.getStock(), p.getImageUrl(), p.isActive(),
                p.getCategory().getId(), p.getCategory().getName(), imgs);
    }
}
```

BREAKING for tests: `ProductMapper` now has a constructor arg. Every `new ProductMapper()` in existing unit tests (`ProductServiceImplTest` line 50) becomes `new ProductMapper("http://localhost:8080")`. This is a known, mechanical edit — call it out in tasks.

### 6.4 `service/impl/ProductServiceImpl.getById()` modification

Single-line swap — the method is ALREADY `@Transactional(readOnly = true)` (line 56), so the LAZY `images` collection loads inside the open transaction (`open-in-view: false` makes this mandatory, not optional):

```java
@Override
@Transactional(readOnly = true)
public ProductResponse getById(Long id) {
    Product product = findOrThrow(id);
    if (!product.isActive()) throw new ResourceNotFoundException("Producto no encontrado: " + id);
    return productMapper.toResponseWithImages(product);   // was: toResponse(product)
}
```
`search()` (line 48) is UNTOUCHED — stays on `productMapper::toResponse` (lean), so the list query never triggers N+1 image loads.

## 7. Frontend

### 7.1 `models/product.model.ts` modification

```ts
export interface ProductImageResponse {
  id: number;
  url: string;          // full serving URL composed by the backend mapper
  displayOrder: number;
  cover: boolean;
}
// ProductResponse gains:
  images?: ProductImageResponse[];   // present only on detail (getById); absent on list
```
`images?` optional — mirrors `@JsonInclude(NON_NULL)` (omitted on list). No frontend HTTP change: `getById()` already returns `ProductResponse`, now richer.

### 7.2 `product-detail.component.ts` — plain CSS carousel

State via a signal-free index field (component already uses plain fields, not signals — match its style):

```ts
currentIndex = 0;

get gallery(): string[] {
  const imgs = this.product?.images;
  if (imgs && imgs.length > 0) return imgs.map(i => i.url);
  return [this.product?.imageUrl ?? PLACEHOLDER_IMAGE];   // fallback: single cover or placeholder
}
get currentImage(): string { return this.gallery[this.currentIndex] ?? PLACEHOLDER_IMAGE; }
get hasMultiple(): boolean { return this.gallery.length > 1; }

prev(): void { this.currentIndex = (this.currentIndex - 1 + this.gallery.length) % this.gallery.length; }
next(): void { this.currentIndex = (this.currentIndex + 1) % this.gallery.length; }
```

Template replaces the single `<img>` with:
```html
<div class="detail__carousel">
  <img [src]="currentImage" [alt]="product.name" class="detail__image" />
  @if (hasMultiple) {
    <button class="carousel__nav carousel__prev" (click)="prev()" aria-label="Anterior">‹</button>
    <button class="carousel__nav carousel__next" (click)="next()" aria-label="Siguiente">›</button>
    <div class="carousel__dots">
      @for (img of gallery; track $index) {
        <span class="carousel__dot" [class.carousel__dot--active]="$index === currentIndex"></span>
      }
    </div>
  }
</div>
```

- Empty/absent `images` → `gallery` falls back to `[imageUrl ?? PLACEHOLDER_IMAGE]`, exactly like the current behavior — zero regression for products without uploaded images.
- `hasMultiple` hides nav/dots when there's a single image.
- Plain CSS + index arithmetic — **NO new dependency** (locked decision). Index wraps with modulo for circular navigation.

## 8. Testing strategy (Strict TDD — RED before GREEN, per layer)

Test command: `cd backend && ./mvnw test` (backend), `npm test` (frontend Jest). PostgreSQL via Testcontainers — NO H2.

### 8.1 Unit — `service/ProductImageServiceImplTest` (JUnit5 + Mockito, no Spring, no DB)

Mocks: `@Mock StorageService`, `@Mock ProductImageRepository`, `@Mock ProductRepository`. Construct impl by hand in `@BeforeEach` (mirrors existing `ProductServiceImplTest` line 50). Pass `"http://localhost:8080"` wherever a URL is needed.

Cases:
- **upload validation**: wrong content-type → `IllegalArgumentException`, `verify(storage, never()).store(any())`; oversize (`file.getSize()` > 5MB) → same; count==10 → `IllegalArgumentException`, no store; happy path → `storage.store` called once, row saved.
- **first-image cover sync**: when `countByProductId==0`, captured saved `ProductImage.isCover()==true` AND captured `Product.getImageUrl()` ends with the stored filename → use `ArgumentCaptor<ProductImage>` + `ArgumentCaptor<Product>` (the §1 pattern).
- **delete non-cover**: cover + `image_url` untouched (`verify` product never re-saved with new url), `storage.delete` called with the row's path.
- **delete cover with survivors**: `findFirst...IdNot...` returns next; captor asserts next `.cover==true` and `product.imageUrl` == serveUrl(next.path).
- **delete LAST image**: `findFirstBy...` returns empty → captor asserts `product.imageUrl == null`.
- **setCover demote+promote**: current cover mock returned → captor asserts old `.cover==false`, target `.cover==true`, `image_url` == target url.
- **reorder**: ids belong to product → displayOrder updated, cover/image_url NOT touched; foreign id in payload → `IllegalArgumentException`.

ArgumentCaptor is the verification lever for every cover-sync assertion (same idiom the codebase already uses in `ProductServiceImplTest`).

### 8.2 Web slice — `controller/AdminProductImageControllerTest` (@WebMvcTest)

Same harness as `AdminProductControllerTest`: `@WebMvcTest(controllers = AdminProductImageController.class, excludeFilters = JwtAuthenticationFilter)` + `@AutoConfigureMockMvc(addFilters = false)`, `@MockBean ProductImageService`. Security boundary (401/403) deferred to integration.

Cases:

Test the **app-level** 400 here (service throws on `file.getSize()`), and verify the true 413 (Spring `MaxUploadSizeExceededException`) in INTEGRATION where the real multipart resolver + configured `max-file-size` are active. Document this split so nobody wastes time forcing 413 in the slice.

**`ImageServeControllerTest`** (`@WebMvcTest(ImageServeController.class)`, `@MockBean StorageService`):
- 200 + body: `when(storage.load("x.jpg")).thenReturn(new ByteArrayResource("bytes".getBytes()))` → assert body + status.
- 404: `when(storage.load(...)).thenThrow(new ResourceNotFoundException(...))`.
- **path-traversal → 400**: `when(storage.load("..%2F..%2Fetc%2Fpasswd")).thenThrow(new IllegalArgumentException(...))` → assert 400 (proves the controller propagates the guard's exception to the handler). The real guard logic is unit-tested directly against `LocalStorageServiceImpl.resolveSafe` and end-to-end in integration.

## 10. ADRs (decisions made THIS phase)

- **ADR-D1 — extension from MIME, not client filename.** `store()` derives the on-disk extension from the validated content-type, never from `getOriginalFilename()`. Rejected: trusting client filename (extension-spoofing + a second traversal vector). Cost: a tiny MIME→ext map.
- **ADR-D2 — guard lives in `resolveSafe`, exposed via `IllegalArgumentException`.** Both `store/load/delete` route through one private `resolveSafe`; traversal throws `IllegalArgumentException` → reuses the EXISTING `handleIllegalArgument` 400 mapping. Rejected: a bespoke exception + new handler (more surface, no benefit). The guard is also unit-testable in isolation.
- **ADR-D3 — `ProductMapper` gains a constructor arg (base-url).** Composing the URL in the mapper keeps the client URL-composition-free and centralizes it with the write-path `serveUrl`. Cost: existing `new ProductMapper()` test call sites must pass the base-url — mechanical, flagged in tasks. Rejected: injecting base-url into the controller and post-processing (leaks a mapping concern into web).
- **ADR-D4 — reorder is cover-neutral.** Reordering never re-elects the cover; only `setCover`/`delete` move it. Rejected: "first after reorder becomes cover" (couples two independent concerns, surprises the admin). Keeps each mutation single-purpose and independently testable.
- **ADR-D5 — demote-before-promote in `setCover`, with a `flush()` fallback.** Set old cover false before new cover true to respect the partial-unique index within one transaction; if integration reveals a flush-ordering race, insert an explicit `repository.flush()` between the two writes. Rejected: dropping the partial-unique index and relying solely on app logic (loses the DB-level invariant).
- **ADR-D6 — 413 tested only in integration; app-level 400 size check in the service.** The `@WebMvcTest` slice can't enforce real multipart size, so the slice asserts the service's own `getSize()` 400 and integration asserts the true 413 from the configured resolver. Rejected: skipping the app-level size check (would make the service non-self-validating and the unit test unable to cover oversize).

## 11. Risks carried forward

1. **CRITICAL — path traversal** on the serve endpoint. Mitigated by `resolveSafe` (canonicalize + `startsWith`), extension-from-MIME, unit + web + integration coverage of the traversal case. Highest-priority test.
2. **Ephemeral uploads dir** in containers — files vanish on restart (academic scope, accepted/documented in proposal). `@TempDir` in integration; `APP_UPLOADS_DIR` lets deploys point at a volume.
3. **Cover/`image_url` sync atomicity** — every cover mutation + url update inside ONE `@Transactional`; partial-unique index is the DB-level backstop. ADR-D5 covers the flush-order edge.
4. **LAZY images read with `open-in-view: false`** — `getById()` MUST stay `@Transactional(readOnly=true)` and use `toResponseWithImages`; `search()` MUST stay on the lean mapper. A regression here = `LazyInitializationException` (detail) or N+1 (list). Integration `GET /api/products/{id}` with images asserts the read path stays open.
5. **`ProductMapper` constructor change** ripples to existing unit-test call sites — mechanical but must not be missed (build break otherwise). Flagged in §6.3 and the affected-files table.

## 12. Verdict

Design complete. Every layer has a concrete contract, the cover-promotion algorithm is fully specified, and the three tricky test patterns (MockMultipartFile, @TempDir + @DynamicPropertySource, 413-vs-400 split) are copy-ready. Ready for `sdd-tasks` once the spec is also in place.
