Caso de negocio
y el problema

Naomi Velasco Moreto
El problema

Krypton vende, pero no puede escalar

Es un retail de tecnología (laptops, smartphones, componentes) con un proceso de venta presencial y manual. Eso le genera:

  • Alcance limitado al horario y la ubicación de la tienda.
  • Inventario propenso a errores, sin control de stock en tiempo real.
  • Imposible atender clientes fuera de la zona.
  • Sin datos consolidados: no hay reportes de ventas ni de productos más vendidos.
La solución

Un e-commerce de n capas en la nube

Digitaliza la venta de extremo a extremo y le da a la administración el control en tiempo real del catálogo, el stock y las ventas.

Objetivo general

Diseñar, desarrollar e implementar una aplicación web de n capas en la nube que automatice el proceso de venta de Krypton, garantizando la persistencia, la seguridad transaccional y la generación de reportes exportables.

Alcance y actores

Qué entra y quiénes participan

Dentro del alcance

Registro/login con JWT · catálogo con búsqueda y filtros · carrito y checkout con control de stock transaccional · panel admin (CRUD) · reportes PDF/Excel · despliegue en la nube.

Fuera del alcance

Pasarela de pago real (el pago se simula) · app móvil nativa · logística de envíos · reseñas y promociones.

Cliente

Se registra, inicia sesión, navega el catálogo, compra y consulta sus pedidos.

Administrador

Gestiona productos, stock, categorías y pedidos; genera reportes de ventas e inventario.

Proceso de negocio central

El proceso de venta en línea

Es el eje del sistema y donde se demuestra la seguridad transaccional: si la confirmación falla, el stock no se descuenta (todo o nada).

Navega el catálogo→ Agrega al carrito→ Checkout→ Genera la orden
(descuenta stock)→ Confirmación→ Admin gestiona y reporta

Modelado en BPMN (docs/krypton-proceso-pedido.bpmn): pool con 2 lanes — Cliente y Administrador.

Capas con interfaces — las dependencias fluyen hacia abajo

Controller

@Entity
  • El controller nunca toca el repositorio: siempre pasa por el service.
  • Las entidades no se exponen en la API: requests/responses son DTOs (mapper Entity↔DTO).

Modelo de datos

8 tablas, 3 decisiones clave

users · categories · products · cart · cart_item · orders · order_items · stock_movement

Snapshot de precio

El order_item congela el precio al comprar; el cart_item usa el precio vivo. Así una orden vieja no cambia si sube el precio.

Stock cacheado + kardex

products.stock es el saldo rápido; stock_movement es el historial (la verdad). Como una cuenta bancaria.

Clave técnica vs. negocio

id (surrogate) para las FKs; sku (natural) legible para el admin. El SKU nunca es FK.

Backend: API,
seguridad y pruebas

Jason Dávila Delgado
API REST

Contrato limpio con DTOs

  • Endpoints REST por recurso: /api/products, /api/cart, /api/orders, /api/admin/**.
  • Entrada y salida siempre por DTO (record), nunca la entidad JPA — el contrato JSON queda desacoplado del modelo.
  • Manejo de errores centralizado (@RestControllerAdvice) → respuestas consistentes (400/401/404/409/422).
Seguridad

Autenticación con JWT

Token JWT

Login devuelve un token firmado (HS512). El front lo manda en cada request (Bearer).

BCrypt

Las contraseñas se guardan hasheadas, nunca en texto plano.

Roles

CLIENTE y ADMIN. /api/admin/** exige rol ADMIN.

Sin token → 401Rol incorrecto → 403Comunicación HTTPS
El corazón del sistema

Checkout = una transacción atómica

Al confirmar la compra, dentro de una sola transacción pasa todo esto — o pasa todo, o no pasa nada (rollback):

  • Se crea la orden y sus ítems (copiando el precio → snapshot).
  • Por cada ítem se registra una SALIDA en el kardex.
  • Se descuenta el stock cacheado.
  • Se vacía el carrito.

Si falla un paso (ej. sin stock), se revierte TODO. Nunca se vende algo sin descontarlo. Esto cumple el pilar de seguridad transaccional.

Pruebas

Construido con Strict TDD

Cada feature del backend se hizo test-first: RED → GREEN → REFACTOR.

Unit

JUnit 5 + Mockito. Lógica del service con el repositorio mockeado.

Web slice

@WebMvcTest + MockMvc. Contrato HTTP: status, JSON, validaciones.

Integración

Testcontainers con MySQL real — paridad con producción (kardex, transacciones).
